# 验证计划

本文件定义 `/verify` 的验证计划制定规则。

## 分组依据

将需要主动验证的 checks 按手段分为以下几组，每组用同一类工具一次性完成：

| 分组 | 适用场景 | 典型 checks |
|------|---------|-------------|
| 静态检查 | 纯代码/配置可判断，无需运行 | 函数签名存在、配置字段正确、文件已创建 |
| 命令验证 | 需要执行命令才能得到结果 | 测试通过、构建无报错、命令输出符合预期 |
| 浏览器验证 | 涉及 UI 渲染、交互或页面行为 | 页面正确显示、按钮点击响应、表单提交流程 |
| Review Helper | 配置了 skill/agent 类型的自定义验证能力，且 helper 描述明确覆盖对应 check | 由 projectSummary 和 helper 描述共同判断是否匹配 |

## 手段选择原则

- 结合 `projectSummary` 判断哪些分组对当前项目类型有意义：web 应用优先考虑浏览器验证，CLI 工具优先命令验证，library 优先测试命令。
- 静态能推断的不必运行，运行能判断的不必开浏览器。
- 每个 check 只归入一种手段，但所有用到的分组都必须出现在计划里，包括静态检查。
- Review helpers 不是默认必选——只在 helper 描述明确覆盖对应 check 时才使用。
- 浏览器验证不依赖 review helpers 是否配置，chrome devtools 工具本身可用时即可选用。
- 一项 check 只归入一个分组；有歧义的归入覆盖面更广的分组。

## 展示格式

向用户展示验证计划，格式如下（有哪些组就列哪些）：

```
验证计划：
【静态检查】
  - T1: <check 描述>
【命令验证】运行 <具体命令>
  - T2: <check 描述>
【浏览器验证】打开 <url>
  - T3: <check 描述>
```

验证计划和过程结果不落盘。
