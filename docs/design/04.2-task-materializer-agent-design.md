# 04.2 Task Materializer Agent Design

`task-materializer` 是 `/do` 首次执行时使用的拆解 agent。它负责把当前任务目录的 `plan.md` 转成 `scripts/run.mjs do materialize` 所需的结构化输入，但不拥有任何状态写入权。

## 目标

`task-materializer` 的目标是把首次物化阶段从 `/do` 主上下文中隔离出来，避免 `/do` 在同一个 session 中同时承担计划理解、任务拆解、上下文确认和后续执行调度。

它只解决一个问题：

- 根据最新 `plan.md` 生成可写入 `task.json` 的初始 `objective` 和 `subtasks[]` 草案。

它不解决：

- 执行 task。
- 恢复执行进度。
- 判断当前应执行哪个 task。
- 校验整个任务是否完成。
- 修改计划目标、范围或验收口径。

## 进入时机

`/do` skill 在满足以下条件时调用 `task-materializer`：

1. 当前项目已初始化。
2. `.my-cc-lite/tasks/` 下存在唯一未归档任务目录。
3. 当前 task 目录存在 `plan.md`。
4. 当前 task 目录还不存在 `task.json`。

如果 `task.json` 已存在，`/do` 不再调用 `task-materializer`，而是直接从已有 `task.json` 恢复执行状态。

## 输入

调用方应提供最小必要上下文：

- 完整 `plan.md`。
- 当前 task 目录路径。
- 可选的项目顶层结构摘要。
- 可选的 `plan.md` 明确提到的文件或目录摘要。
- 当前 `/do` 对首次物化的约束。

默认输入不包含大范围业务代码内容。

允许的有限事实确认包括：

- 读取 `plan.md` 明确提到的文件或目录。
- 查看项目顶层结构，用于判断 task 应按文件、模块、文档或阶段拆分。
- 读取少量已有约定文档，例如 README、AGENTS.md 或相关设计说明。

如果拆解必须依赖大量实现细节，`task-materializer` 不应继续扩大读取范围，而应返回 `coarse_ready` 或 `needs_plan_update`。

## 职责

- 从 `plan.md` 提取 `objective`。
- 将 `Plan` 中的主要工作项拆成 `subtasks[]`。
- 为每个 task 生成稳定、简短的 `id`、`title`、`steps[]` 和 `checks[]`。
- 将需要独立状态、失败重试、跳过或单独委派的工作提升为独立 task。
- 将复杂动作放入当前 task 的 `steps[]`，必要时用嵌套 step 表达子动作。
- 判断拆解是否足够支持后续 executor 执行。
- 在计划缺少目标、范围、执行边界或验收口径时返回可读原因。

## 禁止事项

- 不调用 `scripts/run.mjs do materialize`。
- 不直接创建、修改或删除 `task.json`。
- 不调用 `scripts/run.mjs do update-task`。
- 不修改 `plan.md`。
- 不修改业务代码。
- 不运行测试、构建或格式化命令。
- 不选择当前要执行的 task。
- 不调用 executor、verifier 或 debugger。
- 不判断整个任务是否最终完成。
- 不保存完整 agent 响应、命令日志、diff 或恢复历史。
- 不新增 `task.json` 字段。

## 输出

`task-materializer` 返回结构化结果，由 `/do` skill 决定是否调用阶段脚本写入。

```json
{
  "result": "ready",
  "objective": "Objective snapshot derived from plan.md",
  "subtasks": [
    {
      "id": "T1",
      "title": "Implement the first planned task",
      "steps": ["Read the relevant files"],
      "checks": ["The implementation matches plan.md"]
    }
  ],
  "shouldStopAfterMaterialize": false,
  "reason": ""
}
```

`result` 可选值：

- `ready`：拆解结果可直接交给 `/do` 调用 `materialize`。
- `coarse_ready`：只能形成粗粒度 `subtasks[]`，不应立即物化；`/do` 应先让用户确认是否接受这份粗拆，确认后再调用 `materialize`。
- `needs_plan_update`：`plan.md` 缺少关键目标、范围或验收口径，不应创建 `task.json`。
- `blocked`：缺少必要文件、权限、外部条件或上下文，当前不能可靠拆解。

`shouldStopAfterMaterialize` 只影响 `/do` skill 在物化成功后的流程选择，不写入 `task.json`。`ready` 默认应为 `false`，表示物化成功后继续进入执行链路；`coarse_ready` 在用户确认并完成物化后默认应为 `true`，表示本次只创建 `task.json`，不继续执行 task。

`reason` 应保持简短，用来解释为什么需要停止、回到 `/plan` 或只能粗粒度拆解。

## task 生成规则

- `Objective` 形成 `task.json.objective`。
- `Plan` 的主要编号项通常形成一个 task。
- `Goal` 形成 task `title` 和执行边界。
- `Do` 形成 `steps[]`。
- `Check` 形成 `checks[]`。
- 没有明确 `Check` 时，根据 `Goal` 和 `Do` 生成最小可判断检查项。
- 一个 task 应能被 executor 在局部上下文中独立推进。
- 需要单独失败、重试、跳过或委派的工作不应隐藏在同一个 task 的长 `steps[]` 中。
- 纯说明、背景、非执行性备注不应生成 task。

task id 使用 `T1`、`T2`、`T3` 这类稳定递增编号。不要使用依赖文件名、时间戳或模型判断的动态 id。

如果 `plan.md` 缺少明确 `Objective`，`task-materializer` 必须返回 `needs_plan_update`，不要生成可写入的草案。

## 与 `/do` skill 的关系

`/do` skill 仍是执行阶段主流程所有者：

1. 读取当前状态。
2. 在缺少 `task.json` 时调用 `task-materializer`。
3. 检查 `task-materializer` 输出。
4. 对 `ready` 直接调用 `scripts/run.mjs do materialize`；对 `coarse_ready` 先让用户确认，确认接受后再调用 `materialize`。
5. 对 `needs_plan_update` 或 `blocked` 停止并向用户说明原因。
6. 对 `ready`，在物化成功后由 `/do` 直接选择第一个 `pending` task，并继续进入 executor、verifier 和 debugger 的执行链路。
7. 对 `coarse_ready`，在用户确认并物化成功后停止，提示后续 `/do` 将以已确认的粗粒度 `subtasks[]` 继续执行。

`task-materializer` 只生成 materialize 输入草案。真正写入 `task.json` 的边界仍是 `/do` skill 调用阶段脚本。

## 与其他 agent 的关系

`task-materializer` 只在 `task.json` 不存在时工作。物化成功后，它不再参与后续 `/do`。后续 task 选择由 `/do` 根据 `task.json` 中的状态确定性完成。

### executor

`executor` 执行单个 task。它接收 `task-materializer` 生成并已写入 `task.json` 的 task entry，但不重新拆解整个 `plan.md`。

### verifier

`verifier` 检查当前 task 或最终任务是否满足检查口径。它不新增、删除或改写 `subtasks[]`、`steps[]` 或 `checks[]`。

### debugger

`debugger` 只处理明确失败的最小修复路径。它不参与首次任务拆解。

## 上下文控制

`task-materializer` 可以消耗首次拆解所需的计划上下文，但 `/do` 主流程只接收它的结构化输出和简短原因。

这意味着：

- 读取过的长文档或文件内容不进入后续 executor 输入。
- 后续 executor 只围绕当前 task 再读取必要文件。
- 如果首次拆解已经需要大量代码细节，说明 `plan.md` 的执行边界不足，应回到 `/plan` 或只生成粗粒度 task 后停止。

## 错误处理

`task-materializer` 不返回脚本错误码。它只返回面向 `/do` skill 的拆解决策：

- `needs_plan_update`：提示用户回到 `/plan` 补清目标、范围、执行边界或验收口径。
- `blocked`：说明缺少的文件、权限、外部条件或用户决策。
- `coarse_ready`：说明哪些部分无法可靠细拆，以及为什么需要用户确认后再物化。

阶段脚本错误仍由 `/do` skill 调用 `scripts/run.mjs do materialize` 后处理，例如 `PROJECT_NOT_INITIALIZED`、`NO_ACTIVE_TASK`、`PLAN_NOT_FOUND`、`TASK_ALREADY_MATERIALIZED` 或 `INVALID_INPUT`。

## 验证建议

第一版可以不新增完整测试框架。实现落地时只需要做最小 smoke：

1. 准备一个包含 `Objective`、`Plan`、`Goal`、`Do` 和 `Check` 的 `plan.md`。
2. 调用 `task-materializer` 生成 materialize 输入。
3. 将输出传给 `scripts/run.mjs do materialize`。
4. 确认生成的 `task.json` 结构合法。
5. 确认 `task-materializer` 没有直接写文件。

如果 smoke 需要落盘，放在 `./test/` 下，不混入正式目录。

## 取舍

引入 `task-materializer` 会增加一个 agent 角色，但它换来的是首次物化上下文隔离。`/do` 主流程不再需要在同一个上下文中完成计划拆解和后续执行调度。

这个 agent 不改变状态写入边界，也不引入新的持久字段。它只是把一次性的模型拆解工作从 `/do` 主上下文中移出。
