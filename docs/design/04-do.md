# do module design

`/do` 是分步骤执行阶段。它根据最新 `plan.md` 拆解执行任务，创建或更新 `task.json`，执行变更，并在同一阶段完成 item 级检查。

## 目标

- 从当前 `plan.md` 和已有 `task.json` 中选择或生成待处理 work item。
- 如果当前任务还没有 `task.json`，首次 `/do` 创建它。
- 执行该 item 对应的最小变更。
- 运行与该 item 相关的检查。
- 回写 item 级执行状态。
- 将 item 标记为 completed、failed 或 blocked。

## 输入

- `.my-cc-lite/project.json`。
- 当前任务的 `plan.md`。
- 当前任务的 `task.json`，首次 `/do` 时可以不存在。
- 可选 item id，例如 `/do T2`。

## 执行循环

每个 work item 使用固定循环：

```text
read latest plan.md
-> ensure task.json exists
-> select or create item
-> in_progress
-> execute
-> checking
-> run checks
-> record outcome
-> completed / failed / blocked
```

这意味着 `/do` 内部包含“执行-检查”循环。`/verify` 只做任务级总验证。

## item 选择规则

- 如果用户指定 id，优先选择已有 item；不存在时根据最新 `plan.md` 判断是否可以创建。
- 如果未指定，根据最新 `plan.md` 和已有 `task.json` 选择下一个待执行 item。
- 不自动重试 `failed` item，除非用户明确指定 retry。
- 不处理已 completed 的 item，除非用户明确要求重做。

## 状态变化

```text
pending -> in_progress -> checking -> completed
pending -> in_progress -> checking -> failed
pending -> in_progress -> blocked
```

`blocked` 用于缺少用户输入、权限、外部依赖等无法继续的情况。  
`failed` 用于实现或检查失败，但可以继续修复的情况。

## 检查处理

`/do` 可以运行与当前 item 相关的检查，但 MVP 不维护独立的 `checks.jsonl`、`evidence.jsonl` 或 `changed-files.json`。

检查通过后，将 item 状态回写为 `completed`。检查失败但可以继续修复时，回写为 `failed`。缺少用户输入、权限或外部依赖时，回写为 `blocked`。

## helper 操作

建议 helper 提供：

```text
next-work-item
set-work-item-status <id> <status>
```

## 输出给用户

```text
Completed: T1
Checks:
- npm run check passed
Next: /do
```

如果所有 required item 已完成：

```text
All required work items are complete.
Next: /verify
```

## 待确认

- 是否支持 `/do --retry T1`。
- failed item 再执行时是否保留旧失败摘要。
- `/do` 一次只处理一个 item，还是可以支持 `/do --all`。
